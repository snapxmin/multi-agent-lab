# Claude Code Dynamic Workflow 六大官方标准典型模式
（源自Anthropic官方博客《A harness for every task》，也是工程落地最常用6种，**模式可以自由嵌套组合**；附带拓扑、原理、代码场景、和Subagent/Agent Team的能力边界对比，适配你之前的PPT素材）



## 1. Classify-and-Act 分类路由模式（路由模式）
### 拓扑
输入 → 分类Agent → 根据类别路由至不同任务分支
```
          ┌─分支A Agent
输入 → 分类器 ├─分支B Agent
          └─分支C Agent
```
### 原理
先用低成本Agent（Haiku）对任务/文件/问题做归类，**按类型分发不同处理逻辑**；支持前置路由、也支持后置结果分类汇总。
### 典型工程场景
- 批量工单/Issue分拣：bug、需求、文档问题分流处理
- 代码仓库文件分类：业务代码、配置、测试脚本执行不同扫描规则
- 自动区分简单修复与大型重构，分配不同模型算力

### 原生优势（Subagent难以稳定实现）
分支判断逻辑固化在Runtime脚本，不靠LLM轮次记忆，大批量文件不会丢失路由规则。

## 2. Fan-out-and-Synthesize 扇出-聚合（并行屏障模式，最常用）
### 拓扑
```
规划拆分 → 并行启动大量Worker →【屏障等待全部完成】→ 汇总合成
```
### 原理
大任务拆分为大量互相独立的子任务，并发执行；**Synthesize是屏障（Barrier）**，必须等待所有并行Agent全部结束才进入汇总。每个Worker拥有独立干净上下文，中间结果保存在Runtime，**不污染主会话窗口**。
> 官方标杆案例：Bun百万行代码库跨语言迁移。
### 典型工程场景
- 全仓库批量SDK升级、多文件并行改造
- 遍历src目录所有文件并行安全审计
- 批量接口文档生成、模块依赖梳理

### 和普通Subagent核心区别
普通父子Subagent：所有子任务日志灌入父上下文，规模上涨必然上下文溢出；
DW扇出：中间数据留在脚本变量，仅最终汇总结果回流主对话。

## 3. Adversarial Verification 对抗验证模式（生成-审查）
### 拓扑
```
Worker Agent（产出代码/结论） ←→ 独立校验Agent（对抗审查）
```
### 原理
**每一个执行任务的Agent，配套一个完全隔离的验证Agent**；校验者按照固定标准主动挑漏洞、质疑结论，发现缺陷触发重试。
### 典型工程场景
- 漏洞扫描结果二次核验，消除误报
- 代码重构完成后安全/性能交叉审查
- 高危迁移变更风险校验

> 区分：锦标赛是多个Agent**独立做同一个任务**；对抗验证是「生产者 vs 审查者」固定角色分工。

## 4. Generate-and-Filter 生成+筛选模式
### 拓扑
批量生成多个候选方案 → 筛选Agent打分、去重、过滤 → 输出高质量子集
### 原理
先发散批量产出大量候选，再通过标准化rubric（评判规则）过滤淘汰低质量结果。
### 典型工程场景
- 多种架构方案生成，筛选可行方案
- 多条修复方案生成，择优挑选最小改动方案
- 多版重构思路筛选

## 5. Tournament 锦标赛（淘汰赛择优）
### 拓扑
N个Agent独立使用不同思路完成**同一个目标任务** → 评审Agent两两PK淘汰赛 → 选出最优结果
### 原理
多个Agent互不参考，独立并行解题；评审采用**成对对比**（比单一绝对打分更稳定），逐层淘汰选出最优实现。
### 典型工程场景
- 复杂算法多种实现方案择优
- 疑难bug多种修复思路对比
- 架构方案选型博弈

> 对比Agent Team：Agent Team是大家互相沟通讨论；锦标赛Agent互相隔离、独立产出、事后PK，没有实时交流。

## 6. Loop Until Done 循环直至达标（条件循环模式）
### 拓扑
```
执行任务 → 条件判断
   ↓否（不满足标准）
循环重试
   ↓是（满足停止条件）
结束
```
### 原理
**无固定迭代次数，由脚本条件控制循环**：持续执行任务、校验结果，直到达成停止条件（测试全部通过、不再发现新漏洞、达到最大重试上限）。
### 典型工程场景
- 代码修改 → 跑单元测试 → 失败自动修复循环
- 漏洞持续挖掘，连续多轮无新漏洞才终止
- 迭代优化性能直到指标达标

### 痛点对比
用Subagent模拟循环：重试计数器存在父Agent上下文，长链路极易遗忘计数，无限循环或提前终止；DW把计数器、终止条件写在JS脚本，强制生效。

# 补充：实战高频【组合模式】（工业界最常用，单一模式很少独立使用）
1. **Fan-out + Adversarial Verification**
批量文件并行改造 + 每一份变更自动对抗审查（大规模代码重构标准组合）
2. **Fan-out → Loop Until Done**
并行改造文件，每个文件内部执行「修改-测试-重试」循环
3. **Classify + Fan-out**
先区分文件风险等级，高风险文件启用对抗校验，低风险直接并行改造

# 关键总结（适配你多Agent架构PPT对比页）
1. Subagent / Agent Team：**LLM承担编排决策**，循环、分支、并行依靠Prompt诱导；长链路随机性高、上下文存在硬上限。
2. Dynamic Workflow六大模式：**调度逻辑外置到Runtime JS脚本**，所有分支、循环、并发屏障、重试规则是**确定性代码逻辑**；模型仅作为任务执行者。
